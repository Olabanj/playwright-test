# Time Tracking

## User intent
A company admin/client governs how and when workers record time (time-tracking policies), assigns contracts to those policies, and reviews each contract's effective policy. Workers record time against their contract either live (auto-tracker: clock in, optional pause/resume, clock out, then submit for approval) or as a backdated manual entry (immediately submitted). The client reviews and approves or declines submitted sessions, either individually or in bulk. The platform calculates PAYG payments and overtime from approved sessions, enforces daily working-time limits, and provides a timesheet report and dashboard for visibility. A two-step bulk import lets clients or workers upload a CSV of time entries for mass import. In-platform notifications alert workers and clients to approval events. An audit trail records every policy and entry lifecycle event. Role-based access (worker, client admin, new admin, custom role) controls what each identity can read or mutate. All of the above is also exercised through a web UI with mobile-viewport support.

## Preconditions
- A client account that owns at least one TT policy; the company's TT feature flag is enabled.
- A worker account on at least one ongoing contract (PAYG/per_hour preferred for session flows; fixed_contractor for fixed-contract flows).
- Contracts of every worker type (per_hour/PAYG, fixed_contractor, milestones, direct_employee, eor, per_minute, per_day, per_month) available on the sandbox account for policy-matrix and worker-type tests.
- At least one contract with an assigned TT policy in `ongoing` status (statusId=4) for bulk import and TestEnv-based tests.
- A sandbox user pool of ongoing contracts of each type, leased via `TestEnv` + `FileLock` for parallel isolation; no static `workers.json` or `TEST_CONTRACT_ID` required for most specs (those helpers exist for legacy compatibility only).
- Default policies exist in the company (three system defaults per PD-13481, not deletable, worker-type fields locked).
- No lingering active or paused session for any worker before each test involving clock-in (enforced by `ensureNoActiveSession` / `clearContractSessions`).
- For permissions tests: `CUSTOM_ROLE_EMAIL`/`CUSTOM_ROLE_PASSWORD` env vars set (currently missing in sandbox — those tests are xfail).
- For auto-clockout: `RUN_SLOW=1` env var set (cron-dependent, excluded from normal CI runs).
- For notifications tests: a `TestEnv`-allocated slot with a time-session already created (TestEnv provisions this; test skips gracefully if no slot available).

## Steps (intent only)
1. **Policy management**: client creates a policy specifying timeBasis (flexible / schedule_window with per-day windows / total_hours), workerType (one of 8 types; 7 deployed), overtime config (type daily/weekly, threshold, cap, multiplier — standard or custom decimal), and permission flags (isManualEntryAllowed, isTimeTrackerEntryAllowed, isEditingAllowed, requireReasonOnEdit). Client lists policies (paginated, filterable by timeBasis/workerType/permission flags), reads one by id, checks title availability, updates, and deletes.
2. **Default policies**: client observes three system-default policies (one per worker-type group per PD-13481); confirms none can be deleted; name is editable but workerType field is locked.
3. **Policy wizard scope-2**: client creates/edits policy and observes granularity field per workerType; selects overtime multiplier (standard or custom >0; 0 rejected PD-12914 xfail; negative rejected); changes workerType mid-wizard (resets downstream steps); on incompatible type change with workers a compatibility dialog offers keep or unassign (keepWorkers xfail PD-12929); assigns PAYG contracts to a custom policy.
4. **Policy-contract assignment**: client assigns contracts to policy via PUT (replace entire set; empty array clears all) or PATCH (add/remove individual contracts); resolves effective policy for a contract via policy-by-contract; reassigns a contract from one policy to another; queries contracts with/without assigned policy.
5. **Schedule-window enforcement**: client creates a schedule_window policy with specific weekday windows; manual entry on out-of-window day is rejected with human-readable error including day name (PD-11419, PD-11755); entry on in-window day succeeds; decimal overtime threshold accepted (PD-13703).
6. **Permission-flag enforcement**: worker is blocked from using tracker when isTimeTrackerEntryAllowed=false; blocked from manual entry when isManualEntryAllowed=false (sandbox xfail — boundary not enforced); policy editing flags persist through PATCH.
7. **Worker-type + RBAC**: worker reads own policies and contracts (scoped); client admin has full CRUD; new-admin role has full TT access; custom-role user is denied all TT endpoints (CUSTOM_ROLE env vars required — xfail); unauthenticated requests return 401; worker cannot create policies (sandbox xfail — boundary not enforced).
8. **Feature flag + localization + company-settings permissions**: with TT flag enabled GET /policies and /time-sessions return 200; Accept-Language header in requests yields error messages in the requested language (fr/en); TT Edit role in company settings can access /policies and /time-sessions.
9. **Auto-tracker session**: worker clocks in (no start/end times; server sets startTime; state=active); optionally pauses (state=paused, subsession recorded) and resumes (state=active, new subsession); clocks out (state=completed, endTime set, totalMinutes = worked+paused minutes); submits the completed session (reviewStatus transitions null→pending, making it visible to client for approval). Active session on one contract blocks clock-in on any other contract (worker-scoped one-session constraint). Ending paused session completes directly.
10. **Manual time entry**: worker (or client on behalf of worker) supplies explicit startTime + endTime; session is immediately completed with a single subsession, totalMinutes equal to the slot, pausedMinutes=0; reviewStatus is auto-submitted on creation (no separate submit call needed). Zero-duration and negative-duration entries do not 500. Duplicate slot on same contract is rejected; partial overlap is rejected.
11. **Session edit/delete rules**: editing startTime or endTime of an active (ongoing) session is rejected (PD-11575); editing completed sessions is allowed (PATCH title/note/reason succeeds, PD-11484); worker can delete a pending-approval entry (PD-11576) but not an approved entry; stored UTC time matches the ISO value sent (PD-11703, PD-11800).
12. **Sessions edge cases + states**: resuming an active (non-paused) session rejected; pausing an already-paused session rejected; ending an already-completed session rejected; PAYG clock-out includes payment details in response; stats endpoint requires fromDate; daily stats include submitted session with payment; 10-hour session tracked to full duration.
13. **Daily working-time limit**: attempting a second time entry when daily allowance is fully consumed is rejected (PD-12072, PD-12159); a short first entry that leaves quota headroom allows a second small entry.
14. **Auto-clockout**: backend cron auto-ends sessions exceeding max duration (TC_TT_SESSION_EDGE_010); test is cron-dependent, gated by `RUN_SLOW=1`, and is an xfail because CI cannot guarantee the cron fires within test timeout; kept as a tracked gap (not deleted) per 2026-05-25 parked-test decision.
15. **Approval chain**: client approves single completed+submitted session (reviewStatus→approved); client declines with a reason (reason required — missing reason rejected); client bulk-approves a set of sessions; client bulk-declines with individual reasons; client sends mixed approve+decline bulk request; double-approve blocked; approve→decline transition blocked; decline→approve transition blocked; double-decline blocked; worker cannot approve own session; client-created manual entry starts as pending; bulk with one invalid ID — valid entries still processed; editing an approved session is allowed at API level.
16. **Authorization boundaries**: worker cannot create session on a contract they do not own; worker getContracts returns only own contracts; worker cannot delete approved session (sandbox xfail — boundary not enforced); worker token cannot approve/decline sessions; worker cannot clock in without policy assignment; second concurrent clock-in on same contract rejected (worker-global constraint).
17. **Audit trail**: policy events (CREATE, UPDATE, DELETE, ASSIGN, UNASSIGN) appear in company-level audit history in chronological order, scoped to the company (no cross-company leakage); entry events (SUBMIT, APPROVE, DECLINE, UPDATE, DELETE) appear in contract-level audit history; audit trail is scoped — contract A events do not appear in contract B; worker cannot read company-level or contract-level audit trail data; filtered audit trail returns 200 with valid response shape. (Five async-event xfails in sandbox: ASSIGNED, UNASSIGNED, APPROVED, DECLINED, UPDATED events unreliably ingested under parallel load.)
18. **Payment calculation**: PAYG timeStats.totalAmount is 0 on pending-only days (PD-13692 xfail); overtimeMinutes in timeStats is consistent with paymentStats.overtimeAmount (PD-13693 xfail); daily overtime cap — overtimeMinutes delta does not exceed policy cap (PD-13694 xfail); pending sessions accumulate minutesLogged across multiple sessions; fixed contract stats structure differs from PAYG (no hourlyRate); currency absent from timeStats on pending-only days (PD-13695 xfail).
19. **Payment invariants**: pending PAYG entry has non-zero cost visible in stats (PD-11454); payment currency matches the contract rate currency (PD-11603); totalAmount equals exactly regularAmount + overtimeAmount (PD-11859, PD-11843); minutesLogged reflects billable minutes only (PD-11843).
20. **In-platform notifications**: submitting a session triggers an in-platform notification visible to the client; approved session triggers a notification to the worker; declined session notification carries the reason field; notifications carry a navigation context. Note: email/push delivery is not assertable from CI; coverage is via the in-platform notification feed as a proxy.
21. **Reporting (dashboard stats)**: GET /time-sessions/stats returns paginated structure; per-contract daily stats are within the requested date range; hasPendingTimesessions filter scopes to contracts with pending work; contractType filter returns only matching contracts; fromDate=today returns zero session counts; worker can access own contract's daily stats with correct session counts.
22. **Timesheet report**: POST /reports/preview → 201 (not 200) with pagination meta and up to 10 rows; POST /reports/export → 201 with text/csv Content-Disposition; GET /reports/columns returns 12 default column definitions; custom column selection + ordering is respected; contract-detail columns present when selected (PD-12709); date-range filter narrows results; type filter (regular vs overtime disjoint sets); reviewStatus filters partition data correctly; rejected status filter accepted (PD-12875); workerId/contractType/department/country filters; combined filters intersect correctly; impossible date range returns zero results; export produces valid CSV; unauthenticated → 401; worker role → 403.

24. **E2E flow (combined pipeline)**: full pipeline: create policy → assign contract → worker clocks in → worker clocks out → client approves → confirm approved state (TC_TT_E2E_001, @smoke @critical); flow with pause/resume verifying subsession structure (TC_TT_E2E_002); decline flow confirming reviewStatus=declined (TC_TT_E2E_003).
25. **UI: policy management**: policy list page shows default policy; create policy via wizard (worker-type selection required to advance, all types selectable, end-to-end create); edit existing policy (changes verified); deleted policy card removed from UI; duplicate title rejected by wizard; required-field validation on empty submit; Manage Workers panel opens and lists picker; policy-wizard-scope2 UI: granularity, multiplier options, type-change step reset, compatibility dialog (keep/unassign workers), PAYG assignment list.

## Expected outcome
- Policy create returns 201 with persisted configuration; list returns pagination envelope (`items`, `page`, `limit`, `total_items_count`, `total_pages_count`); update returns 200; delete returns 204 and subsequent GET by id returns 404.
- Default policies cannot be deleted; their workerType field is immutable; three exist (one per worker-type group).
- Title availability returns true for unused titles and false once taken.
- Worker assignment (PUT/PATCH) succeeds and policy `workers` array reflects current contract IDs; unassigned/invalid returns 404 (malformed IDs may return 400/403).
- Auto-tracker sessions transition active → paused → active → completed; completed sessions expose non-null endTime and totalMinutes ≈ workedMinutes + pausedMinutes (endTime=null on completed sessions is a sandbox xfail, TC_TT_SESSION_003).
- Manual entries are immediately completed: single subsession, totalMinutes = slot duration, pausedMinutes = 0.
- Submit transitions reviewStatus null→pending; without submit, ended session is not visible to client.
- Approval/decline transitions are one-way and guarded; bulk operations process all valid items even if one ID is invalid.
- Payment stats for approved PAYG sessions reflect correct totalAmount, overtimeMinutes, and currency; pending-only sessions have 0 totalAmount (currently failing — PD-13692..13695 open bugs).
- Timesheet report endpoints return 201 (not 200); CSV export has Content-Disposition attachment; filters narrow results correctly.
- Bulk import validate returns validEntries/invalidEntries split; execute creates sessions; client uploads auto-approved; worker uploads pending-review.
- UI: wizard creates policies end-to-end; policy list survives reload; approval actions update status immediately; review counter decrements by exactly the number of approved sessions; mobile timer cycle completes successfully; bulk import wizard drives a full happy path.

## Edge cases / variants
- Policy: missing title/timeBasis/workerType → 422; overtime multiplier = 0 → 422 (xfail PD-12914: sandbox accepts it); negative multiplier → 422; incompatible workerType PATCH with workers → 422 (xfail PD-13413); keepWorkers=true retains workers on compatible type change (xfail PD-12929: BE ignores flag); 8th worker type not yet deployed (xfail TC_TT_WTYPE_003).
- Sessions: pause already-paused → rejected; resume active (non-paused) → rejected; end already-completed → rejected; editing active session → rejected; editing endTime of active session → rejected; zero-duration entry does not 500; negative-duration entry does not 500; duplicate slot rejected; partial overlap rejected.
- Daily limit: second entry at exact daily cap boundary rejected; first entry below cap leaves headroom for second.
- Audit trail: async event ingestion is unreliable under parallel sandbox load — 5 audit-trail tests are xfail (TC_AT_004, AT_005, AT_008, AT_009, AT_010).
- Authorization: worker cannot self-approve (TC_TT_REVIEW_009); worker cannot delete approved session (TC_TT_AUTH_003 xfail — sandbox allows it); worker POST /policies not blocked (TC_PERM_031 xfail); worker getTimeSessions returns cross-contract session IDs (TC_PERM_005 xfail); custom-role tests require CUSTOM_ROLE env vars (xfail TC_PERM_020/022/024).
- Bulk import: wrong MIME type → 422; missing column headers → file-level error without crash (PD-13178); mixed valid/invalid rows → correct split with row numbers; weekend date row fails schedule validation.
- Timesheet report: reviewStatus=rejected is a valid filter value (PD-12875); country filter is sent as scalar `country_iso` (not array); impossible date range returns 0 rows and empty CSV.
- UI wizard xfails (34 total): full-page wizard navigation helpers do not advance beyond step 1 (11 wizard-scope2 tests, 2 wizard tests, 3 policy tests); TT contract-tab locator needs re-discovery after DOM change (6 PCT tests); global identity vs pool identity mismatch in approval/worker-entry/time-entry UI tests (14 tests across 4 specs).

## Domain notes
- **Same endpoint, different mode**: `POST /time-sessions` with `startTime`+`endTime` = manual entry; without them = auto-tracker. These are separate flows — mixing them in one spec causes false coverage.
- **Submit step is mandatory for client visibility**: after `endTimeSession`, the worker must call `submitTimeSession` before the session appears in the client's approval queue. This step is undocumented in Swagger. Manual entries do not need a separate submit — they are auto-submitted on creation.
- **Worker-scoped session constraint**: the one-active-session rule is global per worker, not per contract. A stray open session on contract A blocks clock-in on contract B. `ensureNoActiveSession` queries all open sessions (not just for one contract) before each clock-in.
- **Pause/resume body quirk**: `pauseTimeSession`, `resumeTimeSession`, and `endTimeSession` require `id` as a string in the request body (despite being an integer everywhere else).
- **TT /contracts limit cap**: `limit > 50` returns 400. The session helper and TestEnv both cap at 50.
- **Timesheet report is POST→201**: both `/reports/preview` and `/reports/export` are POST endpoints returning 201 (not GET/200 as the name might suggest). Confirmed live 2026-04-20.
- **Probes lane deleted**: this branch deleted both `api/probes/` and `ui/probes/` directories. There are no probe specs remaining; all active coverage is in the `verify/` lane.
- **xfail pattern**: tests with expected failures use `test.fail(true, reason)` — not `test.skip`. They run on every CI invocation and produce signal when the underlying gap closes. 62 total xfails (28 API, 34 UI).
- **Auto-clockout parked**: TC_TT_SESSION_EDGE_010 is gated by `RUN_SLOW=1` and is an xfail. The backend instant-trigger endpoint was proposed by Baha but blocked (no write access to backend repo, Muhammad pushed back). The test must remain as a tracked gap, not deleted, per the 2026-05-25 decision.
- **Workers are assigned by contract ID, not worker/contractor ID**: the policy `workers[]` array holds contract IDs. The TT contracts endpoint exposes `userId` or `contractorId` for the person — use contractId for policy assignment.
- **timeBasis enum discrepancy**: policy create accepts `flexible | schedule_window | total_hours`; legacy policy-by-contract tests assert `timeBasis in ['fixed','flexible']` — this divergence exists in the source and should be reconciled during migration.
- **Payment generation model**: PAYG approved entries create a "Time Entry" payment; other contract types create an "Adjustment" (Addition) payment for overtime only.
- **Bulk import worker ID**: the CSV template uses `Worker ID` which maps to `userId` (from TT contracts endpoint), not `contractId`. The `bulk-import.helpers.ts` documents this: `Worker ID = userId from TT contracts endpoint`.
- **API-first then UI scope decision**: API targets ~100% scenario coverage; UI covers principal flows only (one selection per policy creation wizard, not the full matrix). Per 2026-05-18 scope decision.

## Migration decision
Migrate the time-tracking module into the feature-first layout (`features/time-tracking/`) with all 54 spec files rewritten. Drop the 3-lane structure (probes lane already deleted in this branch). Use `.api.spec.ts` / `.ui.spec.ts` naming, tag-selected runners.

Key rewrites:
- Replace `services/api/modules/time-tracking/TimeTrackingAPI.ts` with a typed `features/time-tracking/client.ts` (typed `ApiResponse<T>`, drop legacy clockIn/clockOut/pauseSession/resumeSession/getSessions surface, keep all active surfaces including bulk-import, timesheet-report, stats).
- Replace `TestEnv` + `FileLock` with a feature fixture (`features/time-tracking/fixtures.ts`) that wraps the same pool-leasing and contract-provisioning logic; or promote `TestEnv` to `core/` as a shared infra fixture since it is domain-agnostic.
- Replace `policy.helpers` and `session.helpers` with typed builder functions in `features/time-tracking/builders/` and a feature fixture that provides policy+session preconditions via Playwright fixture injection.
- Replace workers.json / TEST_CONTRACT_ID coupling with live contract discovery (pattern already established in session.helpers + TestEnv).
- Port all 8 UI POMs to POM-v4 (no `expect` inside page objects; goto stays in page); re-discover the TT contract-tab locator (6 PCT tests currently xfail) and fix the full-page wizard navigation helper (13 wizard tests currently xfail) before marking those tests as passing.
- Carry all xfails across verbatim with their `test.fail(true, reason)` markers and PD ticket references; do not convert to skip or delete.
- Keep auto-clockout test as a tracked gap: `.skip` (or `test.fail(true, ...)`) with inline comment pointing to `docs/30-decisions/2026-05-25-baha-auto-clockout-test-parked.md`.
- Parametrize the policy-contract matrix (currently 18 MATRIX-* tests) into a single data-driven spec.
- Parametrize the invalid-id families (session error-handling, policy-by-contract invalid IDs) into a single loop.
- Merge the `sessions-details/` sub-directory (auto-tracker-sessions, manual-time-entries, session-editing, error-handling) into two flat specs: `sessions.auto-tracker.api.spec.ts` and `sessions.manual.api.spec.ts`.
- Merge approval + authorization + permissions into role-scoped specs following the owner-module rule.
- Resolve the timeBasis enum divergence (create vs read) and the endTimeSession 200/201 inconsistency during rewrite.
- Add the missing submit-after-end coverage in the e2e-flow spec.
- Keep the global identity vs pool identity UI mismatch xfails documented and tracked — requires either migrating UI tests to use the pool-user login (TestEnv provides the contract; the UI test must log in as that same pool user) or decoupling UI auth from TestEnv's pool-user model.
